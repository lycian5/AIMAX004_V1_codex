# COA NEWS 소재 수집 운영 기준

## 1. 목적

이 시스템의 1차 목표는 완성 기사를 자동 생산하는 것이 아니라, 다양한 기사 소재를 넓게 수집하고 수치화해 편집자가 빠르게 판단할 수 있게 하는 것입니다. 원시 수집량은 넉넉하게 유지하되 중복 제거, 출처 신뢰도, 시의성, 공식 근거 여부로 후속 처리 대상을 줄입니다.

## 2. 단일 운영 흐름

1. **기초 수집**: 제목, URL, 출처, 발행 시각, 검색 키워드, 짧은 원문 설명을 저장합니다.
2. **정규화와 중복 제거**: URL과 제목을 정규화하고 동일 사건 후보를 묶습니다.
3. **점수화**: 시의성, 출처 품질, 공식 출처, 교차 확인 수, 키워드 관련성을 계산합니다.
4. **클러스터링**: 같은 사건을 하나의 소재 묶음으로 만들고 대표 제목과 근거 목록을 구성합니다.
5. **리서치 브리프**: 상위 100개 소재를 출처, 사실, 점수와 함께 편집 화면에 노출합니다.
6. **맥락 요약**: 편집자가 선택한 소재에만 AI를 사용해 200~1600자의 판단용 요약을 생성합니다.
7. **기사 작성**: 채택된 소재만 별도의 등록용 기사 준비 화면에서 작성하고 플랫폼에 수동 등록합니다.

원시 수집 단계에서는 OpenAI를 호출하지 않습니다. 소재를 선택하기 전에는 장문 기사도 생성하지 않습니다.

## 3. 수집 채널과 범위

| 계층 | 실행 | 기본 범위 | 역할 |
|---|---|---:|---|
| Vercel 기본 수집 | 매일 06:00 KST | 18개 키워드 | Naver, Google 및 정책 기반의 저비용 기초 수집 |
| VPS Agent Reach | 매일 06:30 KST | 54개 키워드 | Exa, 공식 기관, RSS를 통한 광범위 보강 수집 |
| 소재 정리 | 매일 07:00 KST | 최근 수집분 | 중복 제거, 점수화, 클러스터링, 브리프 갱신 |

Vercel 기본 수집은 핵심 6개와 순환 12개를 사용합니다. Agent Reach는 핵심 12개와 순환 42개를 사용합니다. 활성 키워드가 54개보다 늘어나면 핵심 키워드는 고정하고 나머지 풀을 날짜별로 순환합니다.

YouTube와 GitHub는 인증 및 쿠키 상태가 검증된 뒤 선택적으로 사용합니다. 소셜·영상 자료는 단서로 취급하며, 기사 근거는 원문과 공식 자료로 다시 확인합니다.

한 번의 Agent Reach 실행에서는 RSS 피드를 URL별로 한 번만 내려받아 모든 키워드에 재사용합니다. Exa와 공식 출처 검색은 키워드별로 실행하되 결과 수 상한을 적용합니다.

## 4. 점수와 선별 기준

- 공식 기관 자료와 원문은 높은 출처 점수를 부여합니다.
- 최근 발행, 여러 독립 출처의 일치, 구체적 수치와 일정이 있는 소재를 우선합니다.
- 단일 보도자료 복제, 제목만 다른 중복 기사, 광고성 문서는 감점합니다.
- 공식 확인이 없거나 상충하는 내용은 `needs_verification`으로 표시합니다.
- 수집량이 많아도 동일 사건은 하나의 클러스터로 합쳐 편집 부하를 줄입니다.

대량 수집의 성과는 원시 건수만으로 판단하지 않습니다. 일별로 원시 수집 건수, 고유 URL 수, 중복 제거율, 사건 클러스터 수, 공식 출처 포함률, 브리프 전환율, 소재 채택률을 함께 봅니다.

## 5. 맥락 요약 기준

맥락 요약은 완성 기사나 게시 가능한 원고가 아닙니다. 다음 정보만 간결하게 포함합니다.

- 무슨 일이 있었는지
- 왜 지금 기사 가치가 있는지
- 핵심 인물, 기관, 날짜와 수치
- 추가 확인이 필요한 부분
- 추천 기사 각도

확인되지 않은 사실, 인용문, 수치를 만들지 않습니다. 플랫폼 등록용 기사에는 별도 검토와 편집이 필요합니다.

## 6. 비용 통제

- 원시 수집, 정규화, URL 중복 제거에는 LLM을 사용하지 않습니다.
- AI 호출은 편집자가 선택한 브리프의 맥락 요약과 중요 소재의 추가 검증에만 사용합니다.
- Exa 결과 수, RSS 결과 수, 처리 키워드 수는 환경변수로 제한합니다.
- 긴 본문 수집은 상위 점수 소재에만 수행합니다.
- 정밀 리서치 방식은 법적·정책적 위험이 높거나 다수 출처가 상충하는 선택 소재에만 적용합니다.

## 7. 상태와 권한

- `raw_articles`, `event_clusters`, `editorial_drafts`의 상태는 내부 작업 상태입니다.
- `pending_editor_approval`은 내부 검토대기이며 신문 플랫폼에 실제 등록됐다는 뜻이 아닙니다.
- 플랫폼 등록, 승인, 노출 상태는 플랫폼에서 확인한 값으로 별도 기록해야 합니다.
- 최종 승인과 반려는 편집장 권한이며 상태 변경 시 작업자, 이전 상태, 새 상태, 시각, 메모를 감사 이력에 남기는 것을 운영 원칙으로 합니다.
- 플랫폼에 JSON 또는 CSV 가져오기 기능이 제공되기 전에는 자동 게시를 시도하지 않습니다.

## 8. 자동화 소유권

- Vercel cron과 VPS Agent Reach가 운영 수집의 기준입니다.
- `n8n/workflow_collect.json`과 `n8n/workflow_suggest.json`은 구형 보관본이며 활성화하지 않습니다.
- n8n에서는 Agent Reach runner 호출과 운영 보조만 수행합니다.
- n8n 이미지는 `2.30.7`로 고정하고 업그레이드 전 백업과 복원 시험을 수행합니다.

## 9. 설정 기준 파일

- Vercel 및 로컬 예시: `/.env.example`
- VPS 및 n8n 예시: `/deploy/n8n/.env.example`
- VPS 배포·복원: `/deploy/n8n/README.md`
- 데이터베이스 기준: `/supabase/schema.sql`과 `/supabase/migrations/`

문서나 배포 파일이 서로 다르면 이 문서와 `deploy/n8n` 구성을 우선합니다.
